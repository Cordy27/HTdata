# 公众号新闻接入需求草案

## 状态

- 阶段：已完成实现、CloudBase `005` scale-to-zero 部署、公众号解析、冷启动保护与端到端审计
- 目标开源项目：`wechat-article/wechat-article-exporter`
- 当前新闻系统：华泰互联网组数据门户的 CloudBase 新闻库、关键词筛选、AI 重要度评分与增量快报链路

## 背景与目标

在不重写现有新闻筛选、入库、AI 评分和日报逻辑的前提下，引入指定微信公众号的文章作为新增新闻来源。公众号来源仅负责获取白名单账号及其增量文章，后续处理统一进入现有新闻流水线。

经仓库结构核对，`wechat-article-exporter` 是 Node.js 22 + Nuxt/Nitro 的完整 Web 服务，包含扫码登录、服务端 Cookie/KV 登录态、开放 API、Docker 部署以及 Chromium/Puppeteer 依赖。完整项目不适合原样放入普通 CloudBase 云函数；已确认使用 CloudBase CloudRun 容器承载原服务，不新增云函数调度，统一由现有 GitHub Actions 定时任务触发新闻同步。

## 已确认的产品决策

- 部署形态：CloudBase CloudRun 容器模式部署 `wechat-article-exporter`。
- 调度方式：继续使用现有 GitHub Actions 定时任务，不新增 CloudBase 云函数调度。
- 筛选口径：与现有新闻严格一致，关键词只匹配文章标题；公众号摘要保留入库并供 AI 使用，但不参与第一阶段关键词准入。
- 白名单规模：首期不超过 20 个公众号。
- 登录账号：已有专用公众号账号，登录态失效后允许人工重新扫码恢复。
- AI 与日报：公众号文章与热榜、RSS 合并进入同一个候选池，沿用现有评分和快报逻辑。
- 告警范围：本期不新增飞书、微信、邮件等独立告警，仅使用 GitHub Actions 任务状态和运行日志定位失败。
- 弹性实例：CloudRun 最小实例数设为 `0`；新闻刷新前由同一 GitHub Actions 任务调用健康接口预热，冷启动总预算不超过 120 秒。

## 范围

### 本期包含

- 私有化部署 `wechat-article-exporter`，供内部公众号文章采集使用。
- 配置不超过 20 个公众号的白名单，并将账号名称解析为稳定的公众号 `fakeid`。
- 按计划轮询白名单公众号的增量文章。
- 使用现有 `config/news-sources.json` 中同一套 `keywordGroups` 进行筛选。
- 将命中关键词的文章写入现有 `ht_news_items` 表。
- 将公众号文章与热榜、RSS 新闻一起进入现有 AI 评分和增量快报流程。
- 记录采集、登录态、限流、解析和入库异常，支持运维排查。

### 本期不包含

- 重做现有新闻门户页面。
- 为公众号新闻建立独立的评分模型或独立日报。
- 批量导出 DOCX、PDF、Excel 等面向人工下载的能力。
- 自动化破解或绕过微信登录、风控和访问限制。

## 需求

### 需求 1：部署与访问边界

**用户故事：** 作为新闻系统维护者，我希望公众号采集服务独立部署并通过内部 API 被现有同步程序调用，以便减少对门户主程序的侵入式修改。

#### 验收标准

1. 当公众号采集服务部署完成时，系统应提供健康检查接口和文章列表接口。
2. 当现有新闻同步任务调用公众号服务时，系统应通过服务端鉴权访问，不应在浏览器或静态前端暴露微信登录态、Cookie 或内部 API 密钥。
3. 当公众号服务升级或异常时，现有热榜和 RSS 新闻同步应继续运行，公众号来源失败只记录为单一来源异常。

### 需求 2：公众号白名单

**用户故事：** 作为研究团队，我希望只采集明确批准的公众号，以便控制信息质量、请求量与合规范围。

#### 验收标准

1. 当维护者配置公众号白名单时，系统应保存公众号名称、稳定 `fakeid`、启用状态和最后成功同步时间。
2. 当白名单中存在未解析或失效账号时，系统应跳过该账号并记录可定位的异常，不应阻断其他账号同步。
3. 当公众号不在白名单中时，系统不应主动抓取或入库其文章。
4. 当白名单规模不超过 20 个账号时，系统应支持单次任务逐账号增量同步，无需并行高频抓取。

### 需求 3：增量采集与去重

**用户故事：** 作为研究团队，我希望每次只处理新发布或尚未入库的公众号文章，以便控制调用量并避免重复新闻。

#### 验收标准

1. 当同步任务运行时，系统应从每个白名单公众号的最新文章开始分页，并在遇到已同步游标或超过回看窗口后停止。
2. 当同一公众号文章被重复获取时，系统应按公众号 `fakeid`、文章 `aid` 或规范化文章链接生成稳定标识并幂等写入。
3. 当同一事件被不同新闻来源重复报道时，系统应保留来源信息，并在进入 AI 快报前执行跨来源近似去重，避免日报重复展示。
4. 当文章被删除、封禁或暂时不可访问时，系统应保留历史元数据并标记状态，不应无条件删除已有记录。
5. 当上次任务失败或间隔时间变长时，系统应支持最多拉取接口允许的 20 条最新消息进行补偿，降低漏采风险。

### 需求 4：统一关键词筛选

**用户故事：** 作为研究团队，我希望公众号文章使用与现有新闻完全相同的关键词配置，以便维持统一的选题口径。

#### 验收标准

1. 当公众号文章进入筛选阶段时，系统应读取现有 `keywordGroups`，并与当前热榜、RSS 一致仅匹配文章标题，不应维护第二套公众号专用关键词。
2. 当文章命中关键词时，系统应写入对应标签和命中词字段。
3. 当文章未命中关键词时，系统不应写入正式新闻表，但可在同步运行日志中记录过滤数量。
4. 当关键词配置更新时，后续所有来源应在下一次同步中使用同一版本配置。
5. 当文章摘要或全文包含关键词但标题未命中时，第一阶段系统不应将其入库；若后续需要提高召回率，应统一调整所有来源的匹配字段，而不是只为公众号建立特殊口径。

### 需求 5：统一入库字段

**用户故事：** 作为门户用户，我希望公众号文章与现有新闻以同一数据结构展示，以便继续使用现有页面和接口。

#### 验收标准

1. 当公众号文章入库时，系统应写入现有 `ht_news_items` 表，并将 `source_type` 标识为“公众号”。
2. 当公众号文章入库时，系统应至少保存标题、文章链接、公众号标识、公众号名称、发布时间、摘要、标签、命中词和采集时间。
3. 当现有门户读取新闻数据时，系统应无需新增独立公众号数据接口即可显示公众号新闻。

### 需求 6：AI 评分与增量快报

**用户故事：** 作为研究团队，我希望公众号文章和现有新闻统一评分、统一生成日报，以便获得一个合并后的信息优先级视图。

#### 验收标准

1. 当本轮存在新入库的公众号文章时，系统应将其加入现有 AI 候选池。
2. 当 AI 生成快报时，系统应使用现有提示词、0-100 重要度评分、快讯标题、正文、短信版和跟踪理由字段。
3. 当同一事件来自公众号和其他来源时，系统应优先生成一条合并后的重点信息，并保留可追溯的来源列表。
4. 当公众号服务或 AI 服务失败时，系统应分别记录错误；公众号失败不应影响其他来源入库，AI 失败不应回滚已成功入库的新闻。

### 需求 7：登录态与运维

**用户故事：** 作为系统维护者，我希望能及时发现微信登录态失效并低成本恢复，以便保持定时采集稳定。

#### 验收标准

1. 当微信登录态有效时，系统应将 Cookie 和 token 存放在服务端持久化存储中，不应仅依赖单实例内存或容器本地文件。
2. 当登录态失效或接口返回认证错误时，系统应停止继续请求该来源，并在 GitHub Actions 任务状态和运行日志中明确标记失败原因。
3. 当维护者重新扫码登录后，系统应恢复后续定时同步，无需修改白名单和历史游标。
4. 当同步任务完成时，系统应记录白名单账号数、成功账号数、抓取文章数、关键词命中数、入库数、AI 候选数与异常数。
5. 当 GitHub Actions 调用公众号服务时，系统应使用独立的内部访问密钥；微信侧短期 `auth-key` 不应直接长期保存在 GitHub Secrets 中。
6. 当维护者完成重新扫码登录时，系统应自动更新服务端当前采集会话指针，GitHub Actions 不应因此修改配置或 Secrets。
7. 当登录态失效或公众号来源连续失败时，本期系统应通过 GitHub Actions 任务失败状态和运行日志暴露问题，不要求接入额外消息告警渠道。

### 需求 8：统一 GitHub Actions 调度

**用户故事：** 作为系统维护者，我希望沿用现有定时任务统一处理所有新闻来源，以便只维护一套运行时间、日志和 AI 快报窗口。

#### 验收标准

1. 当现有 GitHub Actions 定时任务运行时，系统应在同一轮同步中依次获取热榜、RSS 和公众号增量文章。
2. 当公众号服务暂时不可用时，GitHub Actions 应继续完成热榜、RSS 入库和可用新闻的 AI 处理，并将公众号异常计入运行日志。
3. 当所有来源完成抓取时，系统应在合并、去重后仅调用一次 AI 快报流程。
4. 当人工触发已有 `refresh_news` 任务时，系统也应同步公众号来源，不应另设独立人工操作入口。
5. 当 CloudRun 已缩容到 0 时，定时任务和 `refresh_news` 应先预热健康接口，再开始公众号抓取；预热只重试网络超时及 HTTP 502/503/504，认证错误不应重试。
6. 当仅人工触发 `force_news_brief` 时，系统应直接使用近期已入库新闻恢复快报，不应因公众号服务预热失败而中断。

## 当前可行性判断

- 已确认：CloudBase CloudRun 容器模式承载项目主体；复用其 Dockerfile，调整 KV 持久化、访问控制和供 GitHub Actions 调用的内部接口。
- 不推荐：把完整 Nuxt/Nitro、Node.js 22、Chromium/Puppeteer 和登录态存储直接打包进普通云函数。
- 可选降级方案：只抽取 `account`、`article`、`download` 等 API 路由做精简 HTTP 服务，但这属于较明显的二次开发，并非原项目低侵入部署。
- 现有门户侧只需新增公众号来源适配器；关键词、CloudBase 表、AI 评分和快报生成逻辑可继续复用。
- 在 20 个以内白名单、每天运行 3 次的情况下，按每个账号每轮读取一页计算约为 60 次文章列表请求/日，规模较低；仍应设置串行或低并发、随机间隔、失败退避和账号级游标。
- 原项目登录态默认保存 4 天；CloudRun 本地文件和单实例内存不能作为可靠持久化方案，需要将 Cookie、token、当前采集会话指针写入仅服务端可读的外部存储。

## 进入技术方案前的确认项

- 用户确认本需求文档后，进入 CloudRun 部署、登录态存储、内部接口、公众号适配器和 GitHub Actions 改造的技术方案阶段。
- 技术阶段解析白名单 `fakeid` 时，如无法唯一确认某个公众号，应向用户请求该公众号的代表文章链接。
