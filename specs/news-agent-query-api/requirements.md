# RSS 与微信公众号聚合查询 API 需求文档

## 状态与边界

- 需求已确认，进入实现。
- 数据范围：配置中的全部 RSS 与白名单微信公众号文章。
- 对外能力：只读检索和批量取数，不提供 AI 整理，不允许触发采集。
- 部署形态：独立 CloudBase HTTP 云函数。

现有流水线在标题未命中关键词时丢弃 RSS/公众号文章，且只保存 Feed 摘要或公众号 digest。本需求将关键词从入库条件改为查询参数，并保存可搜索正文；现有标签、命中词、AI 评分和门户快报逻辑继续保留。

## 本期包含

- 全部 RSS 条目和白名单公众号文章入库。
- 预抓取并保存正文纯文本、可选清洗 HTML、状态、哈希和抓取时间。
- 正文失败不阻止元数据入库。
- 提供来源、搜索、详情、按 ID 批量读取、游标分页和增量读取 API。
- 搜索标题、摘要和正文，支持 any/all/phrase。
- 当前已有 RSS/公众号记录正文回填；默认保留 180 天。
- 独立 Bearer API Key；健康检查除外。

## 本期不包含

- 服务端摘要、聚类、日报、实体提取等整理。
- 外部调用方触发采集。
- 对外暴露热榜、微信 fakeid、登录态、同步游标、Cookie、token 或内部日志。
- WebSocket、SSE、异步 Job 或新的查询 CloudRun。
- 绕过上游认证、风控、版权或访问限制。

## 验收需求

### 1. 全量入库

1. 当 RSS 或白名单公众号标题未命中 keywordGroups 时，系统仍应保存文章。
2. 当文章命中关键词时，系统仍应保存 tags 和 matchedTerms。
3. 当处理热榜时，系统应保持现有关键词/高排名准入行为。
4. 当门户或 AI 快报选择候选时，系统应继续依据现有标签、命中词和评分逻辑，避免全量语料改变展示语义。

### 2. 正文

1. 当 Feed 提供完整内容时，系统应优先使用 Feed 正文。
2. 当 Feed 只有摘要时，系统应尝试从原文 URL 提取正文。
3. 当公众号文章可访问时，collector 应返回清洗正文且不得泄漏凭据。
4. 当正文完整、部分或失败时，系统应分别记录 available、partial 或 unavailable。
5. 当单篇正文失败时，系统仍应保存元数据且不得中断同一来源的其他文章。
6. 当正文更新时，系统应更新正文和哈希并保持稳定新闻 ID。

### 3. 搜索

1. 当未提供关键词时，API 应返回时间范围内全部可见 RSS/公众号文章。
2. 当提供关键词时，API 应支持 title、summary、content 字段和 any/all/phrase 模式。
3. 当提供来源、时间、正文状态、标签、命中词或已有 AI 分数时，API 应组合过滤。
4. 无论客户端参数如何，服务端都应强制只返回 RSS 和公众号。

### 4. 分页和增量

1. 当结果超过单页时，API 应返回不透明 nextCursor，不使用大 offset。
2. 当游标与当前过滤条件不匹配时，API 应返回 INVALID_CURSOR。
3. 当提供 changedAfter 时，API 应按更新时间返回新增或更新记录。
4. 默认每页 30，最大 100。

### 5. 详情和批量

1. 单篇详情应返回正文及明确 contentStatus。
2. 带正文批量读取最多 20 条；仅元数据最多 100 条。
3. compact、standard、full 视图应返回稳定字段集合。

### 6. 安全与部署

1. /health 应允许无鉴权访问；其他请求必须校验 Bearer Key。
2. 数据库和 API 凭据只存于服务端环境变量。
3. 错误应包含稳定错误码和 requestId，不返回堆栈或完整上游错误。
4. HTML 应删除脚本、事件属性和危险 URL。
5. 查询层应使用 HTTP 云函数，通过 scf_bootstrap 监听 9000。
6. API 不可用时，现有新闻同步和门户应独立运行。
7. 数据库升级应采用增量列和索引，保留现有记录。

### 7. 抓取批次增量

1. 当正常同步、公众号恢复同步或其他采集运行首次保存 RSS/白名单公众号文章时，系统应把文章关联到该次同步运行的不可变批次 ID。
2. 当文章仅被重复观察、补全正文、更新 AI 评分或用于强制快报时，系统不得改变文章的首次入库批次。
3. 当调用方未提供 batchId 请求增量接口时，系统应返回最近一个实际产生公开新闻增量的批次，并跳过零增量运行。
4. 当调用方提供有效 batchId 时，系统应返回该批次首次入库的公开新闻；当批次不存在、已过保留期或不含公开新闻时，系统应返回稳定的 BATCH_NOT_FOUND 错误。
5. 当批次内结果超过单页时，系统应返回与 batchId 绑定的不透明 nextCursor；后续翻页请求应固定同一 batchId。
6. 当返回批次元数据时，系统应包含 id、runAt、newCount、status 和 previousBatchId，不得暴露内部 issues、metrics、日志或采集凭据。
7. 当批次运行存在部分来源失败但公开新闻已成功入库时，系统应保留该批次并将公开状态映射为 partial。
8. 当历史记录仍在保留期内且可通过 first_seen_at 与 run_at 唯一匹配时，迁移应回填首次入库批次关联。

## 默认容量

- 保留 180 天。
- 普通搜索默认 30、最多 100。
- 批量元数据最多 100；批量正文最多 20。
- 默认返回纯文本；HTML 仅显式请求。
- 历史正文无法取得时保留元数据和失败状态。
- 批次历史与新闻数据使用相同的 180 天保留期。
